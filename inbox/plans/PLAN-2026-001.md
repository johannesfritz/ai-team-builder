## Product Vision

**Bot the Builder** — A visual no-code builder for Claude Code coworker plugins. Users design agentic workflows on a canvas, test them with dry run simulation, and export production-ready plugin packages.

**What it is now:** A standalone builder tool. Valuable on its own — you can advertise it, demo it, use it yourself.

**What it becomes later:** A platform with marketplace, hosting, and monetization. That comes when the builder has users.

## Why Builder-First

The builder is the product that:
- Solves a real pain you experience daily (hand-authoring `.claude/` directories)
- Requires no marketplace liquidity to be useful
- Can be demoed and advertised immediately
- Generates the content that seeds a future marketplace
- Has zero direct competition

---

## Scope: What We're Building

### In Scope
1. **GitHub OAuth sign-in** — needed for export
2. **Visual plugin builder** (React Flow canvas) — rules, hooks, skills, commands, agents, MCP configs
3. **AI-assisted generation** — describe what you want in natural language, get components generated
4. **Structural dry run** — enter a prompt, see which components fire, estimated token costs
5. **Export** — to GitHub repo (valid `.claude-plugin/` structure) AND downloadable MCPB bundle
6. **Template gallery** — 5-10 starter templates covering common plugin patterns
7. **My Plugins dashboard** — simple list of user's saved plugins
8. **Import from existing `.claude/`** — paste a directory structure, get it loaded into the canvas

### Explicitly Out of Scope (Deferred)
- Marketplace (browse, search, discover other people's plugins)
- Payments / monetization
- Machine Improvement mode (auto-optimize from input/output pairs)
- LLM-powered dry run (actual Claude execution in sandbox)
- Hosted MCP server execution
- Real-time collaboration
- Social features (follow, fork, rate)
- Plugin auto-install into Claude Code

### Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| Frontend | Next.js 16 (App Router), React, TypeScript, Tailwind | One framework, SSR-ready for future marketplace |
| Visual Builder | React Flow (@xyflow/react) | MIT license, saves months vs custom canvas |
| UI Components | shadcn/ui + Geist | Dark mode default, accessible, composable |
| State | Zustand (builder) + React Flow (graph) | Simple, performant, serializable |
| Backend (Web) | Next.js API Routes | Collocated with frontend, Vercel-native |
| Backend (AI) | FastAPI via Vercel Services | Python for Claude API, team strength |
| Database | PostgreSQL (Neon) + Drizzle ORM | Multi-user serverless, JSONB for graph state |
| Auth | NextAuth.js v5 + GitHub OAuth | Free, gives repo API tokens for export |
| Deployment | Vercel (start here, Hetzner option for scale/sovereignty later) | Zero DevOps for MVP |
| AI | Claude API via AI SDK (Sonnet default) | Dog-fooding — Claude understands Claude Code best |

### Key Architecture Decision: Export Formats

The builder exports to **two formats** (from feasibility study insight):

1. **GitHub repo** — Push `.claude-plugin/` directory to a new or existing repo. Users install via `claude plugin add <repo>`.
2. **MCPB bundle** — Download `.mcpb` ZIP with manifest. One-click install in Claude Desktop and Claude Code. No terminal needed.

Both formats generated from the same internal graph representation.

### Plugin Package Structure (Output)

```
.claude-plugin/
├── plugin.json              # Manifest (name, version, description, author)
├── CLAUDE.md                # Project-level instructions
├── skills/
│   └── {name}/
│       └── SKILL.md         # Skill definitions with frontmatter
├── agents/
│   └── {name}.md            # Agent definitions
├── rules/
│   └── {name}.md            # Auto-loaded rules
├── commands/
│   └── {name}.md            # Slash commands
├── protocols/
│   └── {name}.md            # Hook-injected protocols
├── hooks/
│   └── hooks.json           # Hook configurations
├── scripts/
│   └── {name}.{ext}         # Helper scripts
└── .mcp.json                # MCP server configurations
```

## Spikes (Before Building — 2 days)

### SPIKE-1: React Flow Plugin Graph Model (1 day)
**Decision gate.** Can React Flow model Claude Code plugin relationships?

- Build throwaway canvas with 5 node types
- Test: edge validation (hook -> rule OK, rule -> rule blocked)
- Test: serialize graph to valid `.claude-plugin/` directory
- Test: 20+ nodes with custom forms (performance)
- **If fails:** Fall back to form-based editor with optional graph preview

### SPIKE-2: GitHub OAuth + Repo Creation (0.5 day)
- NextAuth.js with GitHub provider
- Verify OAuth scope for repo creation
- Test: create repo, push `.claude-plugin/` structure programmatically

### SPIKE-3: Vercel Services Python Sidecar (0.5 day)
- Minimal `vercel.json` with Next.js + FastAPI
- Test locally (`vercel dev`) and deployed (preview)
- Measure latency, cold start
- **Fallback:** Standalone FastAPI on Hetzner

### WS2: Visual Plugin Builder (3.5 days) — CRITICAL PATH
**Depends on:** WS0, WS1, SPIKE-1
**Agent:** artificial-shadow-dev

**6 custom node types on React Flow canvas:**

| Node | Visual | Key Fields |
|------|--------|------------|
| Rule | Blue card | name, path filter (glob), content (markdown editor) |
| Hook | Orange card | event (PreToolUse/PostToolUse/etc.), matcher, action, `once` flag |
| Skill | Green card | name, description, instructions (markdown), filePattern, bashPattern |
| Command | Purple card | name (/slash-command), prompt template (markdown) |
| Agent | Red card | name, model selector, system prompt, allowed tools |
| MCP Config | Gray card | server name, command, args[], env vars |

**Valid edge types** (enforced by graph validation):
- Hook → Rule (hook injects a rule as context)
- Hook → Skill (hook triggers a skill)
- Agent → Skill (agent uses skills)
- Skill → Command (skill invokes commands)

**Builder UX features:**
- Left panel: component tree (add, reorder, enable/disable)
- Right panel: property editor (forms adapt to selected node type)
- Markdown editor with syntax highlighting and variable autocomplete (`${CLAUDE_PLUGIN_ROOT}`, `${CLAUDE_PROJECT_DIR}`)
- Hook builder: visual When → Condition → Action flow with natural language summary
- Undo/redo (Zustand middleware)
- Auto-save to database
- Canvas minimap + zoom controls

**Key files:**
- `src/app/builder/[id]/page.tsx` — Builder page (client component)
- `src/components/builder/Canvas.tsx` — React Flow wrapper
- `src/components/builder/nodes/` — 6 custom node components
- `src/components/builder/HookBuilder.tsx` — Visual hook configuration
- `src/components/builder/MarkdownEditor.tsx` — Editor with variable autocomplete
- `src/stores/builder-store.ts` — Zustand store
- `src/lib/plugin-schema.ts` — TypeScript types for plugin model
- `src/lib/graph-validation.ts` — Edge validation rules

### WS3: Export Engine (1.5 days)
**Depends on:** WS2, SPIKE-2
**Agent:** artificial-shadow-dev

**Two export paths from the same graph:**

1. **GitHub Export:**
   - Serialize graph → `.claude-plugin/` file tree
   - Create GitHub repo via API (user's OAuth token)
   - Push files with proper directory structure
   - Return repo URL with install instructions

2. **MCPB Bundle Download:**
   - Serialize graph → `.claude-plugin/` file tree
   - Generate `manifest.json` (MCPB format)
   - Package as `.mcpb` ZIP
   - Serve as download

**Serialization mapping:**

| Node Type | Output |
|-----------|--------|
| Rule | `rules/{name}.md` with optional `paths:` frontmatter |
| Hook | Entry in `hooks/hooks.json` (event, matcher, command) |
| Skill | `skills/{name}/SKILL.md` with frontmatter (description, filePattern, bashPattern) |
| Command | `commands/{name}.md` |
| Agent | `agents/{name}.md` |
| MCP Config | Entry in `.mcp.json` |
| (always) | `plugin.json` manifest from plugin metadata |

**Validation before export:**
- All required fields populated
- Hook matchers are valid regex
- Skill filePatterns are valid globs
- No circular edges
- `plugin.json` has name, version, description

### WS4: Dry Run Simulation (1 day)
**Depends on:** WS2
**Agent:** artificial-shadow-dev

**Structural simulation** (no LLM execution):

User enters a sample prompt. The system traces through the plugin lifecycle:

1. **SessionStart** — Which rules auto-load? (check path filters against a sample file tree)
2. **Tool invocations** — User specifies "Claude would use Edit on `src/app.py`" → which hooks fire?
3. **Commands available** — Which slash commands are registered?
4. **Skills matched** — Which skills would match based on filePattern/bashPattern?
5. **Token estimate** — Sum of all injected content (rules + hook outputs + skills)

**Trace output:**
```
Step 1: Session starts
  [✓] Rule "core-patterns" loaded (path: **/*.py matched)
  [✓] Rule "naming-standards" loaded (no path filter — always loads)

Step 2: Claude uses Edit on src/app.py
  [✓] Hook "pre-edit-check" FIRED
      Matcher: Edit matched
      Path: **/*.py matched src/app.py
      Action: Injects code-standards.md (320 tokens)

Step 3: Commands available
  [✓] /type-check registered (450 tokens)

Summary: 3/5 components active, ~770 tokens overhead
```

**Key files:**
- `src/app/builder/[id]/dry-run/page.tsx` — Dry run UI
- `src/lib/simulation/engine.ts` — Trace engine (pure function, no LLM)
- `src/lib/simulation/matchers.ts` — Glob/regex matching for hooks and skills
- `src/components/builder/TraceViewer.tsx` — Step-by-step trace display

### WS5: AI Assist + Import (1.5 days)
**Depends on:** WS0, WS2, SPIKE-3
**Agent:** artificial-shadow-dev

**Two AI features:**

1. **Natural Language → Components:**
   - User describes: "I want a hook that reminds developers about type hints when editing Python files"
   - Claude generates: HookNode (PreToolUse, Edit|Write, **/*.py) + RuleNode (type-hint-standards.md)
   - Components appear on canvas for review/edit

2. **Import from `.claude/` Directory:**
   - User pastes/uploads an existing `.claude/` directory structure
   - Parser reads rules, hooks, skills, commands, agents
   - Generates corresponding React Flow nodes and edges
   - Loads into canvas

**Python sidecar endpoints:**
- `POST /api/ai/generate` — Natural language → component JSON
- `POST /api/ai/suggest` — Given partial plugin, suggest improvements

**Import (TypeScript, no AI needed):**
- `src/lib/import/parse-claude-dir.ts` — Parse `.claude/` structure to graph nodes
- `src/lib/import/detect-edges.ts` — Infer relationships (hook references rule file → create edge)

### WS6: Templates + Dashboard (1 day)
**Depends on:** WS1, WS2
**Agent:** artificial-shadow-dev

**Template Gallery:**
- Grid of template cards with preview
- "Use this template" → loads into builder canvas
- 5-10 starter templates:

| Template | Components | Use Case |
|----------|------------|----------|
| Code Review Standards | 2 hooks, 2 rules | Enforce code quality on edit |
| Git Discipline | 1 hook, 1 rule, 1 command | Commit message standards |
| API Integration | 1 MCP config, 1 skill, 1 command | Connect to external API |
| Multi-Agent Pipeline | 2 agents, 3 skills | Agent team workflow |
| Security Scanner | 2 hooks, 2 rules, 1 protocol | Security validation gates |
| Documentation Enforcer | 1 hook, 1 rule | Require docs before merge |

**Dashboard (simple):**
- "My Plugins" list with name, last edited, component count
- Create new / edit existing / delete
- No social features, no public profiles

### WS7: Testing + Polish (1.5 days)
**Depends on:** WS2, WS3, WS4
**Agent:** qa-engineer + artificial-shadow-dev

- **E2E (Playwright):** Login → create plugin → add components → dry run → export to GitHub
- **Unit tests:** Graph serialization (graph → filesystem round-trip), edge validation, hook matcher logic, dry run engine
- **Export validation:** Generated `.claude-plugin/` structure matches expected format
- **Accessibility:** axe-core automated scans, keyboard navigation through canvas, focus management
- **Polish:** Loading states, empty states, error boundaries, responsive dashboard (builder is desktop-only)
